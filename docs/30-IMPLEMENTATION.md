# 구현 원칙 (Implementation Principles)

> Version 1.0
>
> 이 문서는 구현 원칙을 정의한다.
>
> 구현 언어를 정의하지 않는다.
>
> 코딩 스타일을 정의하지 않는다.
>
> 오직 하나의 질문에 답한다.
>
> **"좋은 엔지니어는 어떻게 구현하는가?"**

---

# 1. 구현은 마지막 선택이다.

좋은 엔지니어는

먼저 구현하지 않는다.

먼저 문제를 해결한다.

구현은

문제를 해결하는 여러 방법 중

마지막 선택지이다.

다음 순서대로 검토한다.

1. 운영으로 해결 가능한가?
2. 설정으로 해결 가능한가?
3. 기존 기능으로 해결 가능한가?
4. 기존 구현을 수정하면 되는가?
5. 새로운 구현이 필요한가?

새로운 구현은

모든 다른 방법이 적절하지 않을 때만 선택한다.

---

# 2. 이해 없는 구현은 하지 않는다.

코드를 보기 전에

문제를 이해한다.

증상이 아니라 원인을 이해한다.

요청이 아니라 목적을 이해한다.

이해하지 못했다면

구현하지 않는다.

---

# 3. 구현 전에 결과를 예측한다.

구현을 시작하기 전에

다음을 먼저 예상한다.

## 예상 결과

이 변경으로 무엇이 달라지는가?

## 영향 범위

무엇이 영향을 받는가?

## 예상 위험

무엇이 실패할 수 있는가?

## 성공 기준

어떤 상태가 되면 성공인가?

## 검증 방법

이 예상을 어떻게 확인할 것인가?

예상 없는 구현은

추측에 가깝다.

---

# 4. 기존 구현을 먼저 이해한다.

새로운 구현보다

기존 구현을 이해한다.

항상 다음 순서대로 탐색한다.

1. 기존 기능
2. 기존 구조
3. 기존 패턴
4. Helper
5. Utility
6. Library

기존 구현으로 해결할 수 있다면

새로운 구현을 만들지 않는다.

---

# 5. Root Cause를 수정한다.

증상을 수정하지 않는다.

공통 원인을 수정한다.

같은 문제가 반복된다면

반복되는 원인을 제거한다.

---

# 6. 가장 작은 변경을 선택한다.

변경 범위는

가능한 한 작아야 한다.

작은 변경은

- 이해하기 쉽고
- 검증하기 쉽고
- 되돌리기 쉽다.

큰 변경보다

작은 완료를 우선한다.

---

# 7. 미래를 구현하지 않는다.

현재 필요한 것만 구현한다.

다음은 구현하지 않는다.

- 언젠가 필요할 기능
- 추측성 확장
- 불필요한 설정
- 사용되지 않는 추상화

필요한 순간에 구현한다.

---

# 8. 단순함을 선택한다.

동일한 결과라면

더 단순한 구현을 선택한다.

짧은 코드보다

읽기 쉬운 코드를 우선한다.

영리한 코드보다

예측 가능한 코드를 우선한다.

---

# 9. 일관성을 유지한다.

프로젝트는

한 사람의 작품이 아니다.

기존 구조와

기존 패턴을 존중한다.

명확한 이유가 없다면

프로젝트의 기존 방식을 따른다.

---

# 10. 예측을 검증한다.

구현은

예측을 확인하는 과정이다.

가능한 방법으로

예상을 검증한다.

예를 들어

- Build
- Test
- 실행
- 로그 확인
- 시뮬레이션
- 코드 리뷰
- 정적 분석

프로젝트에 적합한 방법을 선택한다.

검증 방법은 하나가 아니어도 된다.

중요한 것은

예상을 확인하는 것이다.

---

# 구현 전 체크

□ 문제를 이해했는가?

□ 목적을 이해했는가?

□ 새로운 구현이 정말 필요한가?

□ 기존 구현을 충분히 탐색했는가?

□ 예상 결과를 설명할 수 있는가?

□ 성공 기준을 알고 있는가?

---

# 구현 후 체크

□ 예상한 결과가 실제로 나타났는가?

□ 영향 범위는 예상과 일치하는가?

□ 새로운 문제는 발생하지 않았는가?

□ 변경은 최소 범위였는가?

□ 기존보다 더 단순해졌는가?

---

# 구현 사고 순서

항상 다음 순서대로 생각한다.

문제 이해

↓

목적 이해

↓

결과 예측

↓

기존 구현 탐색

↓

구현 결정

↓

최소 변경

↓

예측 검증

↓

단순화

---

# 하지 않는 것

우리는

많은 코드를 작성하지 않는다.

우리는

복잡한 구조를 만들지 않는다.

우리는

추측으로 설계하지 않는다.

우리는

테스트를 위한 테스트를 만들지 않는다.

우리는

예측 없이 구현하지 않는다.

---

# 최종 원칙

좋은 구현은

코드를 많이 작성하는 것이 아니다.

좋은 구현은

결과를 먼저 예측하고

그 예측을 가장 단순한 방법으로 현실로 만드는 것이다.
