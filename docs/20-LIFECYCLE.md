# 20-LIFECYCLE.md

# 작업 생명주기 (Task Lifecycle)

> Version 1.0
>
> 이 문서는 하나의 작업(Task)이 시작되어 완료될 때까지의 생명주기를 정의한다.
>
> 개발 절차를 설명하지 않는다.
>
> 구현 방법을 설명하지 않는다.
>
> 오직 하나의 질문에 답한다.
>
> **"좋은 작업은 어떻게 시작되고 어떻게 끝나는가?"**

---

# 기본 원칙

작업은

단계를 통과하는 것이 목적이 아니다.

각 단계의 목적을 충족하는 것이 목적이다.

다음 단계는

준비가 되었을 때만 이동한다.

---

# 전체 생명주기

```
Define

↓

Understand

↓

Predict

↓

Implement

↓

Verify

↓

Complete
```

각 단계는

이전 단계의 결과를 기반으로 진행한다.

단계를 건너뛰지 않는다.

---

# 1. Define

## 목적

무엇을 해결해야 하는지 명확히 한다.

## 산출물

- 문제
- 목표
- 완료 조건

## 다음 단계로 이동 조건

□ 무엇을 해결하는지 설명할 수 있는가?

□ 왜 필요한지 설명할 수 있는가?

□ 완료의 기준이 명확한가?

---

# 2. Understand

## 목적

문제를 충분히 이해한다.

증상이 아니라 원인을 이해한다.

요청이 아니라 목적을 이해한다.

## 다음 단계로 이동 조건

□ 원인을 설명할 수 있는가?

□ 기존 구현을 이해했는가?

□ 변경 범위를 이해했는가?

---

# 3. Predict

## 목적

구현 전에 결과를 예측한다.

예상 없는 구현은 하지 않는다.

## 반드시 예상해야 하는 것

### 기능 변화

무엇이 달라지는가?

### 영향 범위

무엇이 영향을 받는가?

### 위험

무엇이 실패할 수 있는가?

### 성공 기준

어떤 상태가 성공인가?

### 검증 방법

무엇으로 성공을 확인할 것인가?

## 다음 단계로 이동 조건

□ 예상 결과를 설명할 수 있는가?

□ 성공 기준이 있는가?

□ 검증 방법이 있는가?

---

# 4. Implement

## 목적

예측한 결과를

가장 단순한 방법으로 구현한다.

구현 원칙은

30-IMPLEMENTATION.md를 따른다.

## 다음 단계로 이동 조건

□ 구현이 완료되었는가?

□ 변경 범위는 최소인가?

□ 구현 의도가 예측과 일치하는가?

---

# 5. Verify

## 목적

예측과 실제 결과를 비교한다.

검증은

방법이 아니라

신뢰를 만드는 과정이다.

검증 방법은

프로젝트 특성에 맞게 선택한다.

예를 들면

- Build
- Test
- 실행
- 로그 확인
- 시뮬레이션
- 정적 분석
- Peer Review

## 다음 단계로 이동 조건

□ 예상 결과와 실제 결과가 일치하는가?

□ 새로운 문제가 발생하지 않았는가?

□ 변경이 신뢰 가능한가?

---

# 6. Complete

## 목적

작업을 종료한다.

종료란

코드를 작성한 것이 아니라

목표를 달성한 상태이다.

## 종료 조건

□ 완료 조건을 충족했다.

□ 사용자 가치가 증가했다.

□ 다음 작업자가 이해할 수 있다.

□ 작업을 종료해도 문제가 없다.

---

# 생명주기의 원칙

단계를 완료하는 것이 아니라

질문에 답하는 것이 중요하다.

질문에 답하지 못하면

다음 단계로 이동하지 않는다.

---

# 작업 중단 원칙

다음 상황에서는

생명주기를 중단한다.

- 요구사항 변경
- 아키텍처 변경
- 보안 영향
- 되돌리기 어려운 변경
- 사람의 판단이 필요한 경우

중단은 실패가 아니다.

올바른 판단이다.

---

# 반복

필요하다면

이전 단계로 돌아간다.

```
Predict

↓

Implement

↓

Verify

↓

Predict
```

반복은

실패가 아니라

학습이다.

---

# 최종 원칙

좋은 작업은

빠르게 끝나는 작업이 아니다.

예측 가능하게 끝나는 작업이다.

작업의 품질은

구현이 아니라

예측과 결과가 얼마나 일치하는가로 결정된다.
